Sometimes, you need to refund a transaction. Blnk allows you to process a refund using the `transaction_id` of the original transaction.

Refunds only happen on transactions that have been applied. This means that the participating balances have been updated with the transaction amount from the original transaction.

When you refund a transaction, Blnk creates a new transaction record and switches the `source` and `destination` balances of the original transaction — debiting the amount from the balance that initially received it and crediting it to the balance that initially sent it.

---

## Applying refunds

Use the **refund-transaction** endpoint and pass the transaction_id of the transaction that you want to refund.


<CodeGroup>
```bash cURL wrap
curl -X POST "http://YOUR_BLNK_INSTANCE_URL/refund-transaction/{transaction_id}" \
  -H "X-blnk-key: <api-key>"
```

```bash cURL (synchronous) wrap
curl -X POST "http://YOUR_BLNK_INSTANCE_URL/refund-transaction/{transaction_id}" \
  -H "X-blnk-key: <api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "skip_queue": true
  }'
```

```javascript TypeScript wrap
const response = await blnk.Transactions.refund(
  'txn_14706192-e53d-43cc-86a1-b8807a09b351',
);
```

```go Go wrap
txn, resp, err := client.Transaction.Refund("txn_14706192-e53d-43cc-86a1-b8807a09b351")
```
</CodeGroup>

Set `skip_queue: true` in the request body to process the refund synchronously instead of queuing it.

## Refunding transactions with multiple sources/destinations

Multiple sources let you send from several sources to one destination at once; multiple destinations let you send from one source to several destinations at once.

Learn more: [Multiple Sources](/transactions/multiple-sources) and [Multiple Destinations](/transactions/multiple-destinations).

<Info>
Refunding the **entire** split in one call works when the original transaction used the **default transaction queue** (`skip_queue: false`). 

Blnk records each leg separately, then links every related transaction with the same `meta_data.QUEUED_PARENT_TRANSACTION` value.
</Info>

When you create a split with the queue, the top-level `transaction_id` in the response is the queued parent. 

That same value is stored on child legs in `meta_data.QUEUED_PARENT_TRANSACTION`. Use it to refund the whole split.

Call the same **refund-transaction** endpoint as above, with that id:

<CodeGroup>
```bash cURL wrap
curl -X POST "http://YOUR_BLNK_INSTANCE_URL/refund-transaction/txn_0b59f6e-6c4a-4efa-915c-526f77ef61ab" \
  -H "X-blnk-key: <api-key>"
```

```javascript TypeScript wrap
const response = await blnk.Transactions.refund(
  'txn_0b59f6e-6c4a-4efa-915c-526f77ef61ab',
);
```

```go Go wrap
txn, resp, err := client.Transaction.Refund("txn_0b59f6e-6c4a-4efa-915c-526f77ef61ab")
```
</CodeGroup>

The response has the same transaction object shape as [Applying refunds](#applying-refunds) above. Blnk refunds all legs in the split from their destinations back to their sources.
